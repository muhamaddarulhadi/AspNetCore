ASP.NET Core Developer Learning Guide

A practical, from-scratch guide to modern .NET and ASP.NET Core — built for junior developers who want to understand not just how but why.

~90 min read 11 sections C# / Razor Pages / EF Core / ADO.NET
01Section

What is .NET?

.NET is a free, open-source developer platform from Microsoft for building many kinds of applications — web, mobile, desktop, cloud, games, IoT, and cloud-native microservices. It includes a runtime that executes code, a set of libraries you write against, and languages that compile down to a common format.

A short history

.NET started in the early 2000s as the .NET Framework, a Windows-only platform. Over the next decade it grew powerful but heavy and tightly coupled to Windows. In 2016, Microsoft released .NET Core, a ground-up rewrite that was open-source, cross-platform, and modular. Starting with .NET 5 in 2020, Microsoft dropped the "Core" branding entirely and unified everything into a single yearly-released product simply called .NET (.NET 5, 6, 7, 8, 9…), with even-numbered releases (6, 8, 10…) marked LTS (Long Term Support).

Note
There is no ".NET Core 4" — Microsoft jumped from .NET Core 3.1 straight to .NET 5 specifically to avoid confusion with the old .NET Framework 4.x line.

Core building blocks

CLR

The Common Language Runtime executes compiled code, manages memory (garbage collection), handles exceptions, and enforces type safety.

BCL

The Base Class Library — thousands of ready-made types for collections, I/O, networking, strings, and more, shared by every .NET app.

JIT Compilation

Your C# compiles to an intermediate language (IL). The Just-In-Time compiler turns IL into native machine code right before it runs.

Runtime

The environment that hosts and executes your app — handling threading, memory, and the JIT — separate from your application code.

How code gets from C# to your CPU

C# / F# / VB.NETSource code
Compiler (Roslyn)Produces IL + metadata
Assembly (.dll)Intermediate Language
CLR + JITRuntime execution
Native machine codeRuns on CPU

Supported languages

.NET is multi-language by design. C# is the primary and most widely used language; F# offers a functional-first alternative; VB.NET is still supported for legacy codebases. All three compile to the same IL and can interoperate freely within one solution.

Cross-platform capability

Modern .NET runs natively on Windows, Linux, and macOS, and on ARM and x64 chips alike. The same codebase can be containerized with Docker and deployed to any cloud provider, which is the biggest practical difference from the old Windows-only .NET Framework.

Why it matters: advantages & use cases

  • Performance — consistently ranks among the fastest mainstream web frameworks in independent benchmarks.
  • Unified platform — one SDK builds web APIs, web apps, desktop apps (WPF/MAUI), and cloud functions.
  • Huge ecosystem — NuGet has hundreds of thousands of packages; strong tooling in Visual Studio, Rider, and VS Code.
  • Common use cases — REST APIs and microservices, enterprise line-of-business apps, e-commerce backends, real-time apps with SignalR, and cloud-native services on Azure/AWS/GCP.

.NET Framework vs .NET Core (Modern .NET)

Understanding this distinction matters because you'll still encounter legacy .NET Framework projects in the wild, while all new development happens on modern .NET.

Capability.NET Framework.NET Core / Modern .NET
Windows supportYesYes
Linux supportNoYes
macOS supportNoYes
PerformanceGoodSignificantly faster (Kestrel, span-based APIs)
DeploymentIIS-centric, Windows onlySelf-contained, containers, any host
Open sourcePartiallyFully open source on GitHub
Cross platformNoYes
Docker supportLimitedFirst-class
Microservices friendlyLimitedDesigned for it
Cloud readyRetrofittedNative (Azure, AWS, GCP)
Future supportMaintenance mode onlyActive yearly releases

Why .NET Core became "modern .NET"

.NET Framework's biggest limitation was that it shipped as part of Windows itself, so every app on a machine shared one framework version — updates could break unrelated apps, and there was no path off Windows. .NET Core solved this by making the runtime an app-local dependency: each app ships (or references) the exact version it was built against. Combined with open-sourcing the entire stack and rebuilding performance-critical pieces from scratch, .NET Core proved so successful that Microsoft retired the "Framework" line and continued innovating only on Core, renaming it simply ".NET" from version 5 onward.

Guidance
Start every new project on the latest LTS release of modern .NET. Only reach for .NET Framework when maintaining an existing legacy system that depends on Windows-only APIs (like WCF or WebForms) that haven't been ported.
03Section

Razor Pages

Razor Pages is a page-focused way of building web UI in ASP.NET Core. Instead of the Controller → Action → View indirection of MVC, each page is a self-contained pair of files: a .cshtml markup file and a .cshtml.cs code-behind class called the PageModel.

Why use Razor Pages?

  • Simpler mental model for page-oriented apps (forms, CRUD screens, dashboards).
  • Each page owns its own logic — easier to navigate than hunting through a shared controller.
  • Still built on the same routing, model binding, and dependency injection as MVC.

Razor Pages vs MVC

Razor PagesMVC
OrganizationPage-centric (view + logic together)Controller-centric (many actions per controller)
Best forForms, CRUD, page-driven appsAPIs, complex multi-action flows
RoutingConvention: folder path = URLConvention: /Controller/Action
TestabilityPageModel is unit-testableController is unit-testable

Project structure

MyApp/
Pages/ routable pages
Index.cshtml + Index.cshtml.cs
Privacy.cshtml + Privacy.cshtml.cs
Shared/ layouts & partials
_Layout.cshtml
_ValidationScriptsPartial.cshtml
_ViewImports.cshtml shared usings/tag helpers
_ViewStart.cshtml sets default layout

_ViewImports.cshtml centralizes @using statements and @addTagHelper directives so every page shares them. _ViewStart.cshtml runs before every page and typically sets Layout = "_Layout";. Partial Views are reusable markup fragments rendered inside a page with <partial name="_Card" />.

The PageModel and handler methods

Every page has a PageModel class deriving from PageModel. Incoming HTTP requests are routed to handler methods named by convention: OnGet handles GET requests, OnPost handles POST, and their async twins OnGetAsync / OnPostAsync are preferred for any I/O-bound work.

Pages/Index.cshtml.cs
public class IndexModel : PageModel
{
    private readonly ILogger<IndexModel> _logger;

    public IndexModel(ILogger<IndexModel> logger)
    {
        _logger = logger;
    }

    public void OnGet()
    {
        // Runs on GET /Index
    }

    public async Task<IActionResult> OnPostAsync()
    {
        if (!ModelState.IsValid)
        {
            return Page();
        }

        // save data...
        TempData["Message"] = "Saved successfully!";
        return RedirectToPage("./Index");
    }
}

Named handlers

A single page can expose multiple POST handlers by suffixing the method name, then targeting them from the form with asp-page-handler.

Named handler example
public async Task<IActionResult> OnPostDeleteAsync(int id)
{
    await _repository.DeleteAsync(id);
    return RedirectToPage();
}
Index.cshtml
<form method="post" asp-page-handler="Delete" asp-route-id="@item.Id">

Binding, TempData, ViewData & ModelState

  • Model Binding — [BindProperty] maps posted form fields directly onto PageModel properties.
  • TempData — survives exactly one redirect, backed by session or cookies; great for "Saved!" flash messages.
  • ViewData — a dictionary that lasts only for the current request, used to pass small bits of data into the view.
  • ModelState — tracks validation results from data annotations; check ModelState.IsValid before acting on posted data.

Routing & URL mapping

By default, a page's file path is its route: Pages/Products/Details.cshtml maps to /Products/Details. You can override this with @page "{id:int}" at the top of the file to capture route parameters that flow straight into handler method arguments via model binding.

04Section

ASP.NET Core Identity

ASP.NET Core Identity is the built-in membership system that manages users, passwords, roles, and claims, and integrates with cookie-based or token-based sign-in.

Authentication vs Authorization


Authentication"Who are you?"

Verifies identity — usually a username/password check that produces a signed-in principal.


Authorization"What can you do?"

Decides whether the now-known identity is allowed to access a resource or action.

In short: authentication happens first and establishes identity; authorization happens second and checks permissions against that identity. A request can be authenticated but still unauthorized (a logged-in user trying to open an admin page).

Core concepts

Login / Logout

SignInManager issues or clears the authentication cookie that represents a session.

Register

UserManager.CreateAsync() creates a new user record with a securely hashed password.

Password Hashing

Identity never stores plain-text passwords — it uses PBKDF2 hashing with a per-user salt.

Claims

Key/value pairs describing the user (email, role, department) attached to their identity.

Roles

Named groups (Admin, Editor, Viewer) that claims and policies can check against.

Cookies

The default carrier for the signed-in session, encrypted and marked HttpOnly for safety.

Security note
Never write your own password hashing. Identity's hasher is battle-tested, salted, and versioned so it can be upgraded without breaking existing accounts.
05Section

Role-Based Authorization & Dependency Injection

Real-world scenario

Consider an internal portal where every user has exactly one role — Admin, Manager, or Employee — stored on their account. Each page declares which roles may open it, and on every request the framework checks the current user's role before rendering the page. An Employee hitting /Payroll/ApproveRaises is redirected away before any payroll data is ever touched.

Request arrivesGET /Admin/Users
Authentication middlewareReads the cookie, builds ClaimsPrincipal
Authorization middlewareChecks role/policy requirement
AllowedPage executes
Page-level role check
[Authorize(Roles = "Admin")]
public class UsersModel : PageModel
{
    public IActionResult OnGet()
    {
        if (User.Identity?.Role == "Admin")
        {
            // proceed
        }
        return Page();
    }
}

Dependency Injection (DI)

Rather than a class creating its own dependencies with new, DI has the framework hand ("inject") them in — typically through the constructor. This is what makes services like an authorization checker swappable, mockable, and centrally configured.

Registering a service
builder.Services.AddScoped<IAuthorizationService, AuthorizationService>();
builder.Services.AddScoped<IUserService, UserService>();
Constructor injection
public class DashboardModel : PageModel
{
    private readonly IUserService _userService;

    public DashboardModel(IUserService userService)
    {
        _userService = userService;
    }

    public async Task OnGetAsync()
    {
        var user = await _userService.GetCurrentUserAsync();
    }
}

Service lifetimes

SingletonOne instance, app lifetime
ScopedOne instance per HTTP request
TransientNew instance every time it's requested
LifetimeLifespanTypical use
SingletonWhole applicationCaching services, configuration objects
ScopedOne requestDbContext, per-request user services
TransientEvery resolutionLightweight, stateless helper services
Why DI beats manual construction
Manually calling new AuthorizationService() hard-wires your class to one concrete implementation, makes unit testing painful (no way to substitute a fake), and scatters configuration logic everywhere. DI centralizes wiring in one place, lets you swap implementations (e.g. a mock in tests) without touching consuming code, and manages object lifetime correctly so scoped resources like a database connection aren't leaked or shared across requests.
06Section

Entity Framework Core

Entity Framework Core (EF Core) is Microsoft's official ORM (Object-Relational Mapper) — it lets you query and update a database using C# objects and LINQ instead of writing raw SQL by hand.

Key building blocks

DbContext

Represents a session with the database — tracks changes and coordinates saving them.

DbSet<T>

A queryable, in-memory-like collection mapped to one database table.

Migrations

Version-controlled, incremental scripts that evolve your schema alongside your model classes.

Navigation Properties

Object references (e.g. Order.Customer) that represent foreign-key relationships.

EF Core execution flow

LINQ querycontext.Users.Where(...)
Query translationExpression tree → SQL
DatabaseExecutes generated SQL
MaterializationRows mapped back to C# objects

Tracking vs AsNoTracking

By default, EF Core tracks every entity it loads so it can detect changes on SaveChanges(). For read-only queries, call .AsNoTracking() to skip that bookkeeping and get a meaningful performance boost.

Method syntax vs Query syntax

Method syntaxQuery syntax
StyleFluent, chained LINQ methodsSQL-like keywords (from/where/select)
Best forMost everyday queries; more operators availableComplex joins that read more naturally in SQL-like form
Method syntax
var activeUsers = context.Users
    .Where(x => x.Active)
    .OrderBy(x => x.Name)
    .ToList();
Query syntax
var activeUsers =
    from u in context.Users
    where u.Active
    orderby u.Name
    select u;

Both compile to the same underlying calls. Reach for method syntax by default — it covers more of the LINQ surface area (e.g. Skip/Take/Any) without mixing syntaxes; drop into query syntax only when a multi-table join genuinely reads more clearly with join ... in ... on ... equals ....

Common LINQ operators

OperatorPurposeSQL equivalent
FirstOrDefaultFirst match, or null/defaultSELECT TOP 1 ...
SingleOrDefaultExactly one match expected, or nullSame, with a uniqueness assumption
AnyDoes at least one row match?EXISTS (...)
CountNumber of matching rowsCOUNT(*)
SelectProject into a shape (DTO)SELECT col1, col2
Join / GroupJoinInner join / grouped (left-like) joinINNER JOIN / LEFT JOIN ... GROUP
GroupByAggregate rows into groupsGROUP BY
DistinctRemove duplicate rowsDISTINCT
Skip / TakePagination — skip N, take MOFFSET ... FETCH NEXT ...
Pagination example
var page2 = context.Products
    .OrderBy(p => p.Id)
    .Skip(20)
    .Take(20)
    .ToList();

Include / ThenInclude

Navigation properties are not loaded automatically. Use .Include() to eagerly load a related collection or reference, and .ThenInclude() to reach one level deeper.

Eager loading
var orders = context.Orders
    .Include(o => o.Customer)
    .Include(o => o.Items)
        .ThenInclude(i => i.Product)
    .ToList();
07Section

ADO.NET

ADO.NET is the low-level data-access API that underlies EF Core itself. It gives you direct control over connections, commands, and result sets — useful when you need maximum performance, complex stored procedures, or fine-grained control that an ORM abstracts away.

Building blocks

SqlConnection

Represents an open connection to the database; always wrapped in using so it's closed/returned to the pool.

SqlCommand

Represents a SQL statement or stored-procedure call to execute against a connection.

SqlDataReader

A fast, forward-only stream for reading rows one at a time.

DataSet / DataTable

In-memory, disconnected representations of one or more result sets.

A SqlParameter binds a .NET value safely into a query or stored procedure call, preventing SQL injection — never build SQL by concatenating strings.

Security
Always use parameterized SqlParameter objects for any value coming from user input. String-concatenated SQL is the single most common cause of SQL injection vulnerabilities.

Execution flow

Open SqlConnection
Build SqlCommand+ SqlParameters
ExecuteReader / NonQuery / Scalar
Map resultsTo objects, DataTable, or DataSet

Stored procedure execution patterns

Click each pattern below to expand a working example.

Uses ExecuteReader() to stream rows and map each one into an object, building up a List<T>.

GetActiveUsers.cs
public List<User> GetActiveUsers()
{
    var users = new List<User>();
    using var conn = new SqlConnection(_connString);
    using var cmd = new SqlCommand("sp_GetActiveUsers", conn)
    {
        CommandType = CommandType.StoredProcedure
    };
    conn.Open();
    using var reader = cmd.ExecuteReader();
    while (reader.Read())
    {
        users.Add(new User
        {
            Id = reader.GetInt32(0),
            Name = reader.GetString(1)
        });
    }
    return users;
}

Same ExecuteReader() pattern, but only reads the first row.

GetUserById.cs
public User? GetUserById(int id)
{
    using var conn = new SqlConnection(_connString);
    using var cmd = new SqlCommand("sp_GetUserById", conn)
    {
        CommandType = CommandType.StoredProcedure
    };
    cmd.Parameters.AddWithValue("@Id", id);
    conn.Open();
    using var reader = cmd.ExecuteReader();
    if (reader.Read())
    {
        return new User { Id = reader.GetInt32(0), Name = reader.GetString(1) };
    }
    return null;
}

ExecuteNonQuery() runs an INSERT/UPDATE/DELETE-style procedure and returns the affected row count.

DeactivateUser.cs
public int DeactivateUser(int id)
{
    using var conn = new SqlConnection(_connString);
    using var cmd = new SqlCommand("sp_DeactivateUser", conn)
    {
        CommandType = CommandType.StoredProcedure
    };
    cmd.Parameters.AddWithValue("@Id", id);
    conn.Open();
    return cmd.ExecuteNonQuery();
}
PurgeExpiredSessions.cs
public void PurgeExpiredSessions()
{
    using var conn = new SqlConnection(_connString);
    using var cmd = new SqlCommand("sp_PurgeExpiredSessions", conn)
    {
        CommandType = CommandType.StoredProcedure
    };
    conn.Open();
    cmd.ExecuteNonQuery();
}

ExecuteScalar() returns just the first column of the first row — ideal for counts, sums, or single IDs.

GetUserCount.cs
public int GetUserCount()
{
    using var conn = new SqlConnection(_connString);
    using var cmd = new SqlCommand("sp_GetUserCount", conn)
    {
        CommandType = CommandType.StoredProcedure
    };
    conn.Open();
    return (int)cmd.ExecuteScalar();
}

A SqlDataAdapter fills a disconnected DataTable in one call — handy when you want an in-memory table without manual row-by-row mapping.

GetOrdersTable.cs
public DataTable GetOrdersTable()
{
    var table = new DataTable();
    using var conn = new SqlConnection(_connString);
    using var cmd = new SqlCommand("sp_GetOrders", conn)
    {
        CommandType = CommandType.StoredProcedure
    };
    using var adapter = new SqlDataAdapter(cmd);
    adapter.Fill(table);
    return table;
}

A stored procedure that returns multiple result sets (e.g. an order plus its line items) can fill an entire DataSet, one DataTable per result set.

GetOrderWithItems.cs
public DataSet GetOrderWithItems(int orderId)
{
    var dataSet = new DataSet();
    using var conn = new SqlConnection(_connString);
    using var cmd = new SqlCommand("sp_GetOrderWithItems", conn)
    {
        CommandType = CommandType.StoredProcedure
    };
    cmd.Parameters.AddWithValue("@OrderId", orderId);
    using var adapter = new SqlDataAdapter(cmd);
    adapter.Fill(dataSet); // dataSet.Tables[0] = order, Tables[1] = items
    return dataSet;
}

Declare a parameter's direction as Output to read a value the stored procedure sets, after execution.

CreateOrder.cs
using var cmd = new SqlCommand("sp_CreateOrder", conn)
{
    CommandType = CommandType.StoredProcedure
};
cmd.Parameters.AddWithValue("@CustomerId", customerId);

var newIdParam = new SqlParameter("@NewOrderId", SqlDbType.Int)
{
    Direction = ParameterDirection.Output
};
cmd.Parameters.Add(newIdParam);

conn.Open();
cmd.ExecuteNonQuery();

int newOrderId = (int)newIdParam.Value;

A stored procedure's RETURN statement (typically a status code) is captured with a parameter direction of ReturnValue.

ValidateOrder.cs
var returnParam = new SqlParameter
{
    Direction = ParameterDirection.ReturnValue
};
cmd.Parameters.Add(returnParam);

conn.Open();
cmd.ExecuteNonQuery();

int statusCode = (int)returnParam.Value;

A SqlTransaction groups multiple commands so they all succeed together (Commit) or all roll back (Rollback) if any step fails.

TransferFunds.cs
using var conn = new SqlConnection(_connString);
conn.Open();
using var transaction = conn.BeginTransaction();

try
{
    using var debit = new SqlCommand("sp_Debit", conn, transaction);
    debit.Parameters.AddWithValue("@AccountId", fromId);
    debit.Parameters.AddWithValue("@Amount", amount);
    debit.ExecuteNonQuery();

    using var credit = new SqlCommand("sp_Credit", conn, transaction);
    credit.Parameters.AddWithValue("@AccountId", toId);
    credit.Parameters.AddWithValue("@Amount", amount);
    credit.ExecuteNonQuery();

    transaction.Commit();
}
catch
{
    transaction.Rollback();
    throw;
}

When to Use Entity Framework vs ADO.NET

CriterionEntity Framework CoreADO.NET
Raw performanceGood, with some ORM overheadFastest — no abstraction layer
FlexibilityConstrained to LINQ-translatable queriesFull control over exact SQL
Stored proceduresSupported but secondaryFirst-class, purpose-built
CRUD development speedVery fast — write C#, not SQLSlower — hand-write everything
Complex reportingCan get awkwardNatural fit for hand-tuned SQL
Large datasetsNeeds care (tracking, paging)Efficient with streaming readers
Bulk operationsWeak by default (needs extensions)Well suited (bulk copy, batching)
MaintainabilitySchema and code stay in sync via migrationsMore boilerplate to maintain
Learning curveModerate (LINQ, change tracking)Lower conceptually, more verbose

Best use cases

Reach for EF Core as your default for line-of-business CRUD apps, rapid feature development, and anywhere schema and code should evolve together through migrations. Reach for ADO.NET (often via a thin repository around a stored procedure) when you need hand-tuned performance on hot-path queries, complex reporting logic that belongs in the database, or bulk data operations at scale. Many real systems use both: EF Core for the majority of the app, ADO.NET for a handful of performance-critical paths.

09Section

Best Practices

Dependency Injection

Depend on interfaces, not concrete classes. Keep constructors small — too many dependencies is a signal to split the class.

Repository Pattern

Wrap data access behind an interface so business logic never talks to DbContext or SqlConnection directly.

Service Layer

Keep business rules out of PageModels/controllers — put them in services that are easy to unit test in isolation.

Authentication & Authorization

Prefer policy-based authorization over scattered role-string checks; centralize rules so they're auditable.

Entity Framework

Use AsNoTracking() for read-only queries; avoid N+1 queries by eager-loading with Include.

ADO.NET

Always parameterize input, always wrap connections in using, and wrap multi-step writes in a transaction.

Error Handling

Catch specific exceptions, never swallow errors silently, and return consistent, safe error responses to clients.

Logging

Use structured logging (ILogger<T>) with meaningful log levels instead of scattering Console.WriteLine.

Async / Await

Use async all the way down for I/O-bound work; never block on async code with .Result or .Wait().

SOLID Principles

Single responsibility, small interfaces, and depending on abstractions keep the codebase testable as it grows.

Takeaway
None of these practices are ASP.NET-specific tricks — they're general software engineering discipline applied consistently. Master them here and they'll carry over to any platform you work in next.

Essential CLI Commands

The dotnet CLI is the tool you'll reach for constantly during day-to-day development — cleaning up build output, compiling your project, and packaging it for deployment. All three commands below are run from your project or solution folder.

dotnet clean

Removes the bin/ and obj/ output from previous builds, so the next build starts from a clean slate.

dotnet build

Compiles your project (and its dependencies) into assemblies, catching compile errors before you run or deploy anything.

dotnet publish

Produces a self-contained, deployment-ready output folder — optimized and trimmed of anything only needed at dev time.

dotnet clean

Deletes the intermediate (obj/) and output (bin/) folders produced by earlier builds. Reach for this when you suspect stale build artifacts are causing odd behavior, or before a fresh CI run.

Terminal
dotnet clean

dotnet build

Compiles the project in the current directory (or the one specified) along with all of its dependencies. By default this builds a Debug configuration; add -c Release for an optimized build.

Terminal
dotnet build
dotnet build -c Release

dotnet publish -c Release

Packages the app for deployment: compiles in Release configuration, resolves NuGet dependencies, and copies everything the app needs to run into a single output folder — ready to copy to a server, drop into a Docker image, or push to Azure.

Terminal
dotnet publish -c Release -o ./publish

The -o ./publish flag is optional and just controls where the output lands; without it, publish output goes to bin/Release/<tfm>/publish/.

Typical local workflow


dotnet cleanWipe old output

dotnet buildCompile & verify

dotnet publish -c ReleasePackage for deployment
Tip
You rarely need dotnet clean for everyday work — dotnet build is incremental and fast. Reach for clean specifically when troubleshooting a build that "shouldn't" be failing, or to guarantee a pristine build in a CI/CD pipeline.

EF Core Migrations to SQL Server

Migrations are how EF Core keeps your database schema in sync with your C# model classes over time. Instead of hand-writing ALTER TABLE scripts, you change your model, generate a migration, and apply it — and EF Core works out the SQL for you.

1. Install the EF Core CLI tool & SQL Server provider

The dotnet ef command is a separate global tool, and your project needs the SQL Server provider package.

Terminal
dotnet tool install --global dotnet-ef
dotnet add package Microsoft.EntityFrameworkCore.SqlServer
dotnet add package Microsoft.EntityFrameworkCore.Design

2. Configure the SQL Server connection string

appsettings.json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=localhost;Database=AppDb;Trusted_Connection=True;TrustServerCertificate=True;"
  }
}
Program.cs
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection")));

3. Create and apply a migration


Model changeEdit an entity class

dotnet ef migrations addGenerates a migration file

dotnet ef database updateApplies it to SQL Server
Terminal
dotnet ef migrations add InitialCreate
dotnet ef database update

migrations add <Name> inspects your DbContext and entity classes, compares them to the last migration snapshot, and writes a new migration file with Up()/Down() methods describing the schema change. database update then executes any pending migrations against the target SQL Server database, creating it first if it doesn't exist yet.

Adding a further schema change

Every time you change a model afterward, repeat the same two-step pattern with a new, descriptive migration name.

Terminal
dotnet ef migrations add AddEmailToUser
dotnet ef database update

Other useful migration commands

CommandPurpose
dotnet ef migrations listShows every migration and whether it's been applied
dotnet ef migrations removeDeletes the most recent, not-yet-applied migration
dotnet ef database update <MigrationName>Rolls the database forward or back to a specific migration
dotnet ef migrations scriptGenerates a plain SQL script — handy for production deployments via a DBA-reviewed script instead of running dotnet ef directly against prod
Production note
Avoid running dotnet ef database update directly against a production SQL Server. Generate an idempotent SQL script with dotnet ef migrations script --idempotent and run that through your normal deployment/release pipeline instead, so schema changes are reviewed and reversible.
